# React → Liquid 水合方案

## 问题背景

`vite-plugin-react-shopify` 在构建时通过 SSG（Static Site Generation）将 React 组件渲染为包含 Liquid 标签的静态 HTML 模板。这些 Liquid 变量（如 `{{ section.settings.title }}`）在服务器端由 Shopify 引擎解析为实际值。

问题在于：浏览器加载页面后，React 执行水合（hydration）时需要将虚拟 DOM 与服务器发送的 HTML 匹配。但 HTML 中的 Liquid 变量已被替换为实际值，而 React 组件在首次渲染时拿到的数据可能不一致（SSG 阶段拿到的只是占位符字符串），导致 hydration mismatch。

---

## 方案概览

整体流程分三个阶段：

```
[构建时 SSG]  →  [服务端 Liquid 渲染]  →  [浏览器 Hydration]
 React 组件         Shopify 引擎             React hydrateRoot()
 生成 Liquid 模板    Liquid → 真实 HTML       注入交互行为
```

核心思路：将动态 Liquid 数据包裹在特殊标记中，水合时从标记中读取服务端实际渲染的数据，注入 React 状态，使虚拟 DOM 匹配实际 DOM。

---

## 一、构建时 SSG（React → Liquid）

### 1.1 组件编译

构建时使用 esbuild 将 React 组件打包为 JS ，在 Node.js 环境中调用 `renderToStaticMarkup()` 执行 SSR。

### 1.2 Settings / Params 追踪

SSR 期间使用 Proxy 拦截对 `useShopifySettings()` 和 `useShopifyParams()` 的访问：

```ts
// SSR 阶段返回 Proxy，拦截属性访问
function useShopifySettings() {
  if (typeof window === "undefined") {
    return new Proxy({}, {
      get(_, key) {
        // 记录被访问的字段
        trackedSettings.add(String(key));
        // 返回 Liquid 变量语法
        return `{{ section.settings.${key} }}`;
      },
    });
  }
}
```

这样在 SSR 阶段，`settings.title` 被渲染为字符串 `{{ section.settings.title }}`，同时 `title` 被记录到追踪集合中。

### 1.3 生成 Liquid 模板

HTML 产物通过 `stripReactLiquidTags()` 后处理移除 `<react-liquid>` 自定义标签，并通过 `unwrapHtmlEntities()` 还原 HTML 实体。最终生成包含以下结构的 Liquid 文件：

```html
{% comment %} IMPORTANT: Auto-generated by vite-plugin-shopify {% endcomment %}

<div
  id="{{ section.id }}"
  data-section-id="{{ section.id }}"
  data-ssg-component="hello-world"
>
  <script type="application/json" data-ssg-props>
  {
    "title": {{ section.settings.title | json }},
    "count": {{ section.settings.count | json }}
  }
  </script>
  <div data-ssg-hydrate>
    <h1>{{ section.settings.title }}</h1>
    <p>Count: {{ section.settings.count }}</p>
  </div>
</div>

{% stylesheet %}
._title_abc123{font-size:2rem}
{% endstylesheet %}

<script type="module" src="{{ 'hello-world.js' | asset_url }}"></script>

{% schema %}
{ ... }
{% endschema %}
```

关键结构：

| 属性/元素 | 作用 |
|---|---|
| `data-ssg-component` | 组件标识符，用于 JS 查找待水合的 DOM |
| `data-ssg-props` | Settings JSON 桥，服务端将实际值注入 |
| `data-ssg-params` | Params JSON 桥（snippet/block 场景） |
| `data-ssg-hydrate` | 水合容器标记，此元素及其子节点由 React 接管 |
| `<script type="module">` | 浏览器 hydration 入口脚本 |

## 二、Hydration 入口（浏览器端）

Vite 为每个组件生成独立的 hydration chunk，入口逻辑（`entries.ts` 的 `load` hook 生成）：

```ts
import { hydrateRoot } from 'react-dom/client'
import { SettingsProvider } from 'vite-plugin-react-shopify/runtime/settings'
import { ParamsProvider } from 'vite-plugin-react-shopify/runtime/settings'
import Component from '~/sections/HelloWorld'

const SELECTOR = '[data-ssg-component="hello-world"]'

function hydrate(el) {
  // 找到水合容器
  const h = el.querySelector(':scope > [data-ssg-hydrate]')

  // 读取 props bridge
  const propsEl = el.querySelector(':scope > script[data-ssg-props]')
  const props = propsEl ? JSON.parse(propsEl.textContent) : {}

  // 读取 params bridge
  const paramsEl = el.querySelector(':scope > script[data-ssg-params]')
  const params = paramsEl ? JSON.parse(paramsEl.textContent) : {}

  // 用实际数据重新渲染并水合
  hydrateRoot(h,
    createElement(SettingsProvider, { value: props },
      createElement(ParamsProvider, { value: params },
        createElement(Component)
      )
    )
  )
}

// 初始扫描 + Shopify 编辑器事件监听
scan(document)
document.addEventListener('shopify:section:load', (e) => scan(e.target))
document.addEventListener('shopify:section:unload', (e) => sweep(e.target))
```

### 2.1 水合数据流

```
                构建时                         服务端                       浏览器
          ┌──────────────┐          ┌─────────────────────┐       ┌──────────────────┐
settings  │ Proxy 拦截    │          │ Liquid 引擎解析       │       │ JSON.parse()    │
  ──────► │ 返回 Liquid   │ ──────► │ {{ settings.x }}    │ ────► │ 获取真实值      │
          │ 占位符字符串   │          │ 替换为实际值           │       │                │
          └──────────────┘          └─────────────────────┘       └──────┬───────────┘
                                                                         │
                                                                   ┌─────▼───────────┐
                                                                   │ SettingsProvider │
                                                                   │ 传入 React 树    │
                                                                   └─────────────────┘
```

### 2.2 Liquid 组件

对于需要在 JSX 中嵌入条件渲染等逻辑的场景，提供 `<Liquid>` 组件。SSR 阶段渲染为 `<react-liquid>` 自定义标签（防止 React 转义），后处理步骤剥离标签，保留原始 Liquid 代码：

```tsx
// SSR 端渲染
export function Liquid({ code }) {
  return createElement("react-liquid", {
    dangerouslySetInnerHTML: { __html: code },
  })
}

// 客户端端直接跳过渲染（Liquid 已在 HTML 中）
export function Liquid() {
  return null
}
```

使用示例：

```tsx
export default function Section() {
  const s = useShopifySettings<{ show_title: boolean }>()
  return (
    <div>
      <Liquid code={`{% if ${s.show_title} %}`} />
      <h1>Title</h1>
      <Liquid code="{% endif %}" />
    </div>
  )
}
```

## 三、Shopify 编辑器事件

| 事件 | 行为 |
|------|------|
| `shopify:section:load` | Section 被添加/修改后重新渲染，重新 hydrate |
| `shopify:section:unload` | Section 被删除/即将重绘，unmount React 根节点 |

编辑器修改 setting → Shopify 重新渲染 Section HTML（带着新的 `data-ssg-props`）→ 触发 `unload` → 触发 `load` → 用水合读取新 JSON bridge 并 hydrate。

## 四、已知问题与待解决

### 4.1 富文本 / HTML 内容处理

**现状**：当前方案仅处理纯文本内容（`{{ section.settings.title }}`）。对于富文本字段（如 `type: "richtext"` 的产品描述），Liquid 渲染后包含 HTML 结构，而 JSON bridge 和 Proxy 追踪无法正确处理这些 HTML 内容。

**影响**：
- `richtext` 类型的 setting 值本身包含 HTML 标签，通过 `| json` 过滤器输出到 JSON bridge 时会被转义
- `data-ssg-hydrate` 容器内的 HTML 结构与水合后的虚拟 DOM 不匹配
- 需要特殊处理 HTML 类型的内容，区分"数据"与"结构"

**建议方向**：
- 对于 HTML 内容字段，渲染时使用 `<Liquid>` 组件直接嵌入，绕过 React 的文本内容处理
- 客户端水合时使用 `dangerouslySetInnerHTML` 设置 HTML 内容，避免 React 的安全警告
- 对富文本内容进行 XSS 过滤（如 DOMPurify）后再设置

### 4.2 data-liquid-key 使用内容哈希

**现状**：如果使用递增数字作为 Liquid 占位符的标识（如 `data-liquid-key="1"`、`data-liquid-key="2"`），不同构建之间可能因为组件添加/删除/排序变化导致 key 不一致，从而引发 hydration 异常。

**建议**：使用内容哈希（如 SHA256 前 8 位或基于内容特征的短哈希）作为 `data-liquid-key`：

```ts
import { createHash } from "node:crypto"

function contentHash(content: string): string {
  return createHash("sha256").update(content).digest("hex").slice(0, 8)
}
```

这样相同内容始终产生相同 key，跨构建稳定。

### 4.3 dangerouslySetInnerHTML 与 XSS

**现状**：水合富文本内容时，如果直接通过 JSX 表达式 `{richtext}` 渲染包含 HTML 的字符串，React 会将其转义为纯文本并报警告。

**解决方案**：在客户端 hydration 中，对于 HTML 类型字段使用 `dangerouslySetInnerHTML`：

```tsx
function RichTextContent({ html }: { html: string }) {
  return <div dangerouslySetInnerHTML={{ __html: html }} />
}
```

**XSS 风险**：富文本内容可能来自用户输入（商家在 Shopify 编辑器中编辑），需在以下环节防护：

1. **服务端**：Shopify 的 `richtext` 类型已做基础过滤（限制允许的标签），但不完全可靠
2. **Hydration 端**：在设置 `dangerouslySetInnerHTML` 前进行客户端清理

推荐方案：在读取 `data-ssg-props` 中的 HTML 字段后，通过 DOMPurify 过滤：

```ts
import DOMPurify from "dompurify"

function cleanHTML(dirty: string): string {
  return DOMPurify.sanitize(dirty, {
    ALLOWED_TAGS: ["b", "i", "em", "strong", "a", "p", "br", "ul", "ol", "li", "span"],
    ALLOWED_ATTR: ["href", "target", "rel"],
  })
}
```

### 4.4 其他注意项

- **`||` 兜底值问题**：SSR 阶段 Proxy 返回的 Liquid 字符串永远 truthy，`||` 兜底无效。不要在水合路径中使用 `||` 提供默认值
- **CSS Module hash 差异**：SSG 预渲染的 HTML 使用原始类名，水合后才替换为 hash 类名。接受短暂样式跳跃或使用全局 CSS
- **无效 HTML 嵌套**：`<ul>` / `<table>` 等有严格子元素约束，浏览器可能"修复" DOM 导致与虚拟 DOM 不一致
